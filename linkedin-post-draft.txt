Why would OpenAI buy a Python package manager?

OpenAI acquired Astral (uv, ruff) in March 2026. Three months earlier, Anthropic acquired Bun. Claude Code had just hit a $1B annual run-rate.

These aren't developer relations plays. Both companies are buying the execution layer underneath their coding agents.

The logic:
- Codex generates code in 2 seconds. pip takes 20 seconds to install dependencies. The model isn't the bottleneck — the tooling is.
- uv is 10-100x faster than pip. Ruff lints CPython in under 300ms. Bun's test runner is 11x faster than Jest.
- At millions of agent sessions per week, per-operation savings compound into material compute reductions.

But speed is half the story. The other half: reliability.

Agents fail when environments aren't reproducible. Non-deterministic dependency resolution, environment drift between local and CI, flaky test setups — these are all tooling problems, not model problems. No amount of training data fixes non-deterministic package resolution.

uv's --frozen flag installs exactly what's pinned. bun ci fails if dependencies drift. Deterministic environments are what make autonomous agent loops viable.

OpenAI is assembling a Python stack: Astral (uv, ruff, ty) + Promptfoo (security scanning) + Rockset (codebase indexing).
Anthropic is assembling a JS/TS stack: Bun (runtime + bundler + test runner in one binary) + Vercept (autonomous UI interaction).

Two companies. Two ecosystems. Same strategy: own every layer between the model and running, tested code.

The company that controls the fastest, most deterministic path from prompt to running, tested code has the stickiest product.
